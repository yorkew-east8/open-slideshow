# write-slides — 编写新幻灯片

本仓库是一个 Markdown → Slidev 播放的演讲工具。用户要新写一份幻灯片时，**不要直接开始铺内容**，先按下列步骤把方向对齐，再产出 `.md`。

## 前置：读 know-how

开始前**先读** `user-slides/AGENTS.md`，遵守其中的：
- 合法性（frontmatter + `---` 分页，否则首页校验会拦截）
- 文件骨架与 Slidev 语法速查（布局、Mermaid、代码高亮、备注）
- 文件名 kebab-case + `.md`，主题**不要硬写死**（首页风格选择会注入 `theme` 覆盖）
- 资源放 `apps/player/public/` 或用外链，不放超大文件

## 步骤 1：拿到大纲或梗概

- 如果用户**已经给了**大纲/梗概/主题 → 直接进入步骤 2。
- 如果**没有** → 先问清楚：这次要讲什么？给谁讲？目的是什么？让用户先给一句话到一段话的梗概。

## 步骤 2：明确写作三角（在产出大纲前对齐）

在列大纲前，先用一两句和用户确认这三件事——它们决定后续每一页的取舍：

1. **听众是谁**：画像（角色、背景、已知/未知边界）→ 决定讲述方式与深度。
2. **他们需要什么**：他们关心什么、痛点是什么 → 用来抓注意力、制造共鸣。
3. **我们希望他们做什么**：本次演讲的最终目的（被说服 / 采取行动 / 学会某事）→ 融合前两者形成说服力。

> 若用户梗概里已含上述信息，直接归纳复述确认，不必逐条追问。

## 步骤 3：对齐大纲，逐页明确边界

基于步骤 2，输出一份**逐页大纲**：每一页的标题 + 内容边界（这页讲什么、不讲什么、用什么形式：文字/图/表/代码/Mermaid）。

主动找出并和用户确认：
- **模糊**：表意不清、概念未定义的地方。
- **歧义**：同一句话可被理解成多种含义的地方。
- **明显不合理**：与听众画像不符（过深/过浅）、与目的无关、逻辑跳跃。

页面之间应符合**金字塔写作原则**：上层是结论，下层是支撑该结论的论据/事实，前后页是「推荐关系」（上一页抛出问题 → 下一页给出回答），而不是松散罗列。

反复迭代大纲**直至用户明确认可每一页的标题与边界**，再进入步骤 4。**不要**在边界未定前就开始写正文。

## 步骤 4：填充内容

按已确认的逐页边界，完善每一页。原则：

- **一页一主题**，目的清晰；页面之间逻辑连贯（金字塔原则）。
- **能用图/表就不用文字**：流程用 Mermaid，对比用表格，结构用图。
- **整体风格一致**：图表与图片风格统一，文字精简，单页不堆砌（Slidev 单页不自动分页）。
- **按需联网找资料/图片**：从权威来源获取（优先一手资料、官方文档、知名媒体），**验证来源可靠性，不要轻信小道消息/营销软文**；引用数字与结论时标注出处。外链图片注意可访问性与版权。
- 严格遵守 `user-slides/AGENTS.md` 的语法与资源规范。

## 步骤 5：落地到文件

- 默认输出到 `.env` 里 `SLIDES_DIR` 指定的目录（仓库当前为 `user-slides/`）。
- 除非用户**明确指定**了别的路径，否则用默认目录。
- 文件名 kebab-case + `.md`，例如 `tech-talk.md`。
- 写入后告诉用户文件路径，以及可在首页（容器内 :8080）选中该文件后播放。
